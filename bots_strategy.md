> 完整的演变过程（为什么这么改、中间否决过什么方案）保留在下面按日期排列的正文里，不删，

`_choose_move` 是四层级联，从高到低优先级依次是：

1. **强制换人**（精灵阵亡/被弹出）——`_best_switch` 选分最高的候选。
2. **Ribombee 固定开局脚本**（回合 ≤ 2，仅 Ribombee 首发时触发）——第 1 回合放黏黏网，第 2 回合
   按"对面是不是草系 + 换 Kyogre 划不划算"决定放麻痹粉还是切换。
3. **硬短路斩杀检测**（`_find_ko_move`）——常规招式能斩杀就直接用；常规扫不到时，若太晶还没用过，
   假设太晶化后再扫一遍（v3 新增），扫到能斩杀的就太晶化 + 出招。**太晶只在这一种"能把打不死变成
   能斩杀"的场景触发，不做防御性太晶**。
4. **通用单回合价值打分**（兜底，覆盖以上都不触发的情况）——对所有合法招式/换人分别打分，选分数
   最高的。打分维度：招式伤害（威力 × STAB × 属性克制 × 期望命中次数）、换人安全度（防御端风险 vs
   攻击端收益）、铺垫招式（强化价值 × 这回合扛不扛得住，看双方速度决定用哪套数值算）、状态招式、
   钉子/减益壁（延迟收益打分，按"对面/己方还剩几只精灵可能换上来"算）、Taunt 反叠盾（对面已经有
   正面能力加成才触发）、团队预览克制对照表加成（开局算好的"这只对面精灵我方谁克制"）。

**当前默认参数（消融验证过、暂无更优替代）**：`ENABLE_HARD_KO=True`、`ENABLE_RIBOMBEE_OPENING=True`、
`SETUP_BOOST_WEIGHT=30`、`SWITCH_DEFENSE_WEIGHT=60`、`SWITCH_COST=25`，外加 v2/v3 新增的
Taunt/克制表/延迟收益/太晶几个常量，均已确认插值不敏感或已是最优。2026-07-24 用修复后的消融工具
（issue #14）全量重跑，`baseline` 打出 15/15 满分，`max_damage-uber` 超时问题也确认是测试工具
bug 不是决策逻辑问题，详见 `Ablation_Study.md`/`Experiment_Log.md`。

已知还没做的：防御性太晶、Arceus-Ghost/Eternatus 的专门角色定位（issue #7）、`_survivability_factor`
门槛精细化（issue #7 附带发现，铺垫/Taunt 目前几乎从不触发）。

## 补充设计（跟 Claude 讨论后细化，2026-07-21）

### 前提：对手 teampreview 是随机的

验证了 `poke_env` 源码：`Player`/`SimpleHeuristicsPlayer` 都没覆写 `teampreview`，默认走
`random_teampreview`——**15 个 bot 每局开场用哪只精灵都是随机抽的**，不能假设固定对手。
所以规则必须是运行时根据 `battle.opponent_active_pokemon` 判断的通用逻辑，不能针对"对面大概率开谁"写死分支。
我方 `teampreview` 固定 `/team 1`，单打里只有首发重要（Ribombee），后续换人顺序不受限制。

### 整体决策架构：硬短路规则 + 单回合价值打分

不是纯 if-elif 优先级链，也不做多回合搜索/minimax（对面是不会博弈的固定 bot，没必要），而是：

1. **极少数"结果确定"的场景直接硬短路**，不进打分系统：目前只有一条——**这回合有招式能斩杀对面就直接用**（多个能斩杀的招式里选更安全的，比如没反作用力的优先）。硬短路的意义是让这类确定性结论在代码/报告里能一句话讲清楚，不用绕到"因为分数 0.97 > 0.82"。
2. **其余所有场景（打哪个招式、要不要换人、要不要铺垫）统一走单回合价值打分**，对当前所有合法动作分别算分，选分数最高的。
   - 打分只看**这一回合**的结果，不做多回合推演。
   - 换人评分：从**防御端**（对面招式打在候选换人身上的属性倍率，越低越安全）和**攻击端**（候选人招式打在对面身上的属性倍率，越高越好）两个方向综合打分，谁分高换谁。这个函数是通用的，不专门为 Kyogre 写死，只是 Kyogre 常规情况下大概率算出来分数最高。

### Ribombee 开局分支（固定脚本，不进打分系统）

Ribombee 首发是刻意设计好的固定开局脚本，优先级高于通用打分框架：

```
第 1 回合：无脑放 Sticky Web（黏黏网，减对面换人速度）

第 2 回合：
  对面是草系？（Stun Spore 对草系必定免疫，不浪费回合尝试）
    → 是：直接 U-turn 换 Kyogre
    → 否：查"换人评分"——现在换 Kyogre 划算吗？
          → 划算（被克制）：U-turn 换 Kyogre
          → 不划算：放 Stun Spore（麻痹粉）
```

"被克制"这个词统一复用换人评分函数的结果，不单独定义一套标准，避免开局和中后期用两套不一致的判断。

U-turn 换上 Kyogre 后，唤雨（Drizzle）自动生效，配合雨天加成的瀑布攀登（120 等效威力）是既定的强开局连招。

### Focus Sash 的注意事项

Focus Sash 只在"满血状态下被斩杀"时触发。第一回合场上还没有钉子，这局稳吃保命没问题。
但 Ribombee 是虫/妖精，对岩石系是弱点——**如果后期又把 Ribombee 换回场，且对面已经放过隐形岩，
一上场先掉 25%，Focus Sash 直接作废**。规则上明确"Ribombee 只在开局用一次，不主动换回来"。

### 铺垫招式（剑舞/冥想）的安全触发条件

铺垫本身不产生这回合的伤害，价值来自"以后打得更狠"，所以核心问题是"以后能不能兑现"。
简化公式：`铺垫分数 = 强化本身的价值 × 这回合能扛住对面攻击的把握`。

**关键补充：判断"扛不扛得住"之前，必须先比双方速度，因为谁先手决定了要用哪套数值去算：**

- **我方更快**：铺垫这个动作这回合真的会生效（不会被对面抢先打死导致落空）。
  - 如果铺垫招式本身也加防御向数值（冥想同时加特防、铁壁加防御），"扛不扛得住"可以用**铺垫后**的数值去算对面这波攻击，判断更宽松。
  - 如果铺垫只加攻击向数值（剑舞只加攻击，不加防御），"扛不扛得住"还是要用**铺垫前**的防御数值算，跟没铺垫时一样。
- **对面更快**：对面这回合会先出手，打在我方**还没来得及铺垫**的身上——如果这一下直接把我方打死/打残，铺垫等于完全没兑现就没了（比正常输出还差，因为一点伤害都没造成）。这种情况下"扛不扛得住"必须用**铺垫前（当前）**的数值去算，标准要收紧，不能用"铺垫后应该能扛住"这种事后才成立的状态来判断，因为对面先手时那个状态根本不会出现。

完整链条：**先比速度谁快 → 根据先后手决定用哪套数值判断"扛不扛得住" → 扛得住才给铺垫打分，扛不住铺垫分数直接归零**（此时正常输出/换人的分数会更高，铺垫选项被自然淘汰）。

### v3：太晶化（Tera）MVP（2026-07-23 实现，issue #5）

v1/v2 阶段延后不做（当时的理由：临场判断"这一下能不能把输的局面掰回来"涉及变量太多）。v3 正式
开始后，范围收紧成最有把握的单一场景，仿照硬短路 KO 的哲学：**只在"太晶化能把一个打不死的招式
变成能斩杀"时触发**——`_find_ko_move` 先按常规属性扫一遍，扫不到才假设太晶化成配置的太晶属性
（`_tera_power_score`，STAB/属性跟随太晶后的状态重算）再扫一遍，扫到了就太晶化 + 出这个招。
明确不做防御性太晶（需要预判对手，超出系统能力）。

**2026-07-24 消融验证通过**：`baseline` 3 次重跑 `mean_beaten=14.3/15`，跟太晶化之前的历史
最佳基线完全一致，零回归。翻 `decision_log` 确认机制真的生效了（不是死代码）——45 场对局里
触发 5 次太晶断死，全部是科依大/无极汰那（两只太晶属性都是火）在对手已经掉血、常规打不死的
场景下触发，没有误判斩杀线或开局乱用太晶的迹象。分数没有提升（设计阶段就说好不指望这个大幅
提升胜场），但新增了一个之前完全没有的决策维度且验证是安全的。过程中排查出两个真实 bug
（Judgment 太晶后属性判断反了；`mon.tera_type` 这个 poke_env 属性不是"配置的太晶属性"而是
"协议已揭示的太晶属性"，我方自己出战精灵太晶前一直是 None，改成自己维护 `OUR_TERA_TYPES`
查表）。完整设计/实现/debug/结果记录见 `Experiment_Log.md` 2026-07-23～07-24 五条太晶相关
条目。
